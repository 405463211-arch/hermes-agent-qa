## 5. 升级过程中的"血泪教训"（**下次升级必读**）

### 5.1 `git reset` 抢资源事故

- **现象**：合并到一半 `git status` 显示干净，工作区改动全没了
- **根因**：`hermes_cli/main.py::_stash_local_changes_if_needed`（行 5667-5710）在合并冲突期间检测到 unmerged + dirty，主动调用裸 `git reset` 清空 index → merge 状态被销毁
- **触发链**：用户/IDE 后台触发 `hermes update`（行 6777）→ 该函数在 `git checkout main` 之前自动 stash → 内嵌 `git reset`
- **指纹**：`git reflog --date=iso | grep "reset: moving to HEAD"`
- **预防方案**：见 `.cursor/commands/sp_sync_upstream_release.md` Step 0 + 检查清单

### 5.2 测试文件硬编码本地路径

- **现象**：本地版的 `test_cli_approval_ui.py` 含 `~/PyCharmMiscProject/hermes-agent-main/venv/bin/python`
- **风险**：合到上游会暴露作者本地结构（profile 名、用户名、目录布局）
- **应对**：合并前用 grep 扫一遍 `tests/` 目录的所有 `/Users/`、`PyCharmMiscProject` 等本机特征字符串，统一替换为 `/tmp/sample-*` 占位符

### 5.3 上游做了"扩大匹配"的 bugfix，反而误伤本地用例

- **典型例子**：`_needs_deepseek_tool_reasoning` 把 `"deepseek" in model` 当 thinking 判据
- **教训**：每次接受上游的"判定逻辑/正则/匹配集合"扩大改动，都要立刻跑回归测试（特别是 V3/V4/non-thinking variant 这类边缘场景）
- **最低线**：本次合并后该测试用例已写好回归防护

### 5.4 Change-detector 测试是合并地狱

- **典型例子**：`test_matches_previous_manual_builtin_tool_set`（断言 27 个工具名完全相等）
- **本地策略**：anchor 模式（只断言关键工具子集存在）
- **守则**：见 `AGENTS.md` "Don't write change-detector tests" 铁律——它直接体现在本次合并 §3.8 的决策
