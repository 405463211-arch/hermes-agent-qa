## 5. 升级过程中的"血泪教训"（**下次升级必读**）

### 5.1 `git reset` 抢资源事故

- **现象**：合并到一半 `git status` 显示干净，工作区改动全没了
- **根因**：`hermes_cli/main.py::_stash_local_changes_if_needed`（行 5667-5710）在合并冲突期间检测到 unmerged + dirty，主动调用裸 `git reset` 清空 index → merge 状态被销毁
- **触发链**：用户/IDE 后台触发 `hermes update`（行 6777）→ 该函数在 `git checkout main` 之前自动 stash → 内嵌 `git reset`
- **指纹**：`git reflog --date=iso | grep "reset: moving to HEAD"`
- **预防方案**：见 `.cursor/commands/sp_sync_upstream_release.md` Step 0 + 检查清单

### 5.2 测试文件硬编码本地路径

- **现象**：本地版的 `test_cli_approval_ui.py` 含 `~/PyCharmMiscProject/hermes-agent-main/venv/bin/python`
- **风险**：合到上游会暴露作者本地结构（profile 名、用户名、目录布局）
- **应对**：合并前用 grep 扫一遍 `tests/` 目录的所有 `/Users/`、`PyCharmMiscProject` 等本机特征字符串，统一替换为 `/tmp/sample-*` 占位符

### 5.3 上游做了"扩大匹配"的 bugfix，反而误伤本地用例

- **典型例子**：`_needs_deepseek_tool_reasoning` 把 `"deepseek" in model` 当 thinking 判据
- **教训**：每次接受上游的"判定逻辑/正则/匹配集合"扩大改动，都要立刻跑回归测试（特别是 V3/V4/non-thinking variant 这类边缘场景）
- **最低线**：本次合并后该测试用例已写好回归防护

### 5.4 Change-detector 测试是合并地狱

- **典型例子**：`test_matches_previous_manual_builtin_tool_set`（断言 27 个工具名完全相等）
- **本地策略**：anchor 模式（只断言关键工具子集存在）
- **守则**：见 `AGENTS.md` "Don't write change-detector tests" 铁律——它直接体现在本次合并 §3.8 的决策

### 5.5 静默合丢（**最隐蔽的合并坑** — 2026-05-07 发现）

**现象**：当上游"重写整段代码"时（例如 `_BUNDLED_PLUGIN_DIR: Path = ...` 被改成 `def get_bundled_plugins_dir() -> Path: ...`），git 三向合并会把它当作"上游单方面动作"——**完全不标 `<<<<<<<` 冲突**，但你 baseline 里加在那段的代码会被一并吞掉。

**指纹**：
- `git show --cc <merge_commit> -- <file>` 输出**空**（如果有冲突会有输出）
- `git diff <baseline>..HEAD -- <file>` 显示删除了 baseline 加的具体行
- 测试用 `monkeypatch.setattr(plmod, "<symbol>", ..., raising=False)` 拼写一个**HEAD 不存在**的属性 → `raising=False` 静默吞掉错误，测试 monkeypatch 完全失效但不报错

**v0.12.0 实测**：`hermes_cli/plugins.py` 9 行 `auto_enable` 实现被静默丢（块详情见 §9.3），3 个 `TestPluginAutoEnable` 测试同时失败。

**预防**：每次合并完都跑 `bash scripts/audit_merge_loss.sh`（详情见 sp_sync_upstream_release.md Step 8.5）。

def get_bundled_plugins_dir() -> Path:

| 根因 | 修法地点 | 决策时间 |

## 10. `flush_memories` 误判复盘（2026-05-07，§3.6.1 + §3.6.6 撤销）

> **本节为合并后第二次复盘**——发现 §3.6.6 决策（"取本地"）是**错的**，已于 2026-05-07 撤销。
> **错误**：保留了 248 行 `flush_memories` 函数体，但 0 个调用方，留下死代码 + 文档漂移。
> **修复**：删 def + `_memory_flush_min_turns` config + `cli-config.yaml.example` 注释 + 加 anchor 测试。

### 10.1 错误是怎么发生的

合并 §3.6 时只读了 `def flush_memories` 的冲突标记，决定"取本地保留函数体"。**没有检查同函数的 call sites**。结果：

- baseline `6198fe35f` 在 `_compress_context` + `cli.py:new_session` + `cli.py` 退出 cleanup 共 **3 处** `self.flush_memories(...)` 调用
- 上游 PR #15696（commit `ea01bdceb`，2026-04-25）**整体删掉了 def + 3 处 call sites**
- 上游的 3 处删除 hunk 跟 baseline 的 3 处加入位置**完全重合** → git 三向合并把它当作"上游单方面动作"，**不标 `<<<<<<<` 冲突直接采纳上游版本**
- 合并者只看到 `def` 的冲突标记，没意识到 call sites 已经被合并器静默吞掉
- 结果：函数体留下来，但**没有任何东西调用它**——纯 248 行死代码

### 10.2 这是 §5.5「静默合丢」模式 D 的变种

但**不是合丢业务实现**——是**合丢调用方**。模式 D 在 §5.5 描述时只覆盖了"上游重写整段 → baseline 加的代码被吞"的场景；本案是"上游删除整段 → baseline 加的调用方被吞"，是同一类问题的另一面。

**新发现的检测指纹**：
- `git diff <baseline>..HEAD -- <file>` 显示**只删不加**（baseline 加的调用方在 HEAD 不在了）
- `git diff <upstream-parent>..HEAD -- <file>` 显示**只加 def 不加调用**（HEAD 相对上游多了 def 但没 call sites）
- 全仓 `grep "<func>("` 仅命中 `def <func>(` 一行

### 10.3 上游 PR #15696 的删除理由（必须读）

Teknium 在 commit message 列了 4 条删除理由，**其中第 3 条直接命中 `AGENTS.md` P0 铁律**：

1. **Pre-dates the background review loop** — 现在 background memory review 每 10 轮跑一次（本地 fork 也有，`memory.nudge_interval`），频率远高于"压缩前一次性"
2. **Blocking and synchronous** — flush 在压缩前阻塞用户响应（5-30s）
3. **Cache-breaking** ⚠️ — flush 临时把 toolset 换成只有 `memory` 一个工具，**毁掉 prompt cache**——直接违反 `AGENTS.md`：
   > Do NOT alter past context, change toolsets, or rebuild system prompts mid-conversation.
4. **Redundant** — background review 在 live conversation context 里跑、写同一个 memory store、不毁 cache，覆盖了 flush 想做的所有事

也就是说：**本地 fork 即使把 call sites 接回去，也直接违反自家铁律**。这进一步确认 §3.6.6 的"取本地"决策错了——应该跟 PR #15696 一起取上游。

### 10.5 给下一次合并者的指导（**重要**）

1. **§3.6 任何"取本地"决策**：除了看 `def` 的冲突标记，**必须**额外执行：
   ```bash
   # 找出本地相对上游多出来的 def
   diff <(git show <upstream-parent>:run_agent.py | grep -E "^    def ") \
        <(git show HEAD:run_agent.py | grep -E "^    def ")

   # 对每个本地多出来的 def，验证 HEAD 里有没有调用方
   grep -n "self\.<func_name>\|agent\.<func_name>" run_agent.py cli.py hermes_cli/ gateway/
   ```
   如果 def 存在但调用方为 0 → **要么补回 call sites，要么删 def**。**绝不能"先留着再说"**。

2. **任何"本地独有"的方法**：合并时**必须**对照 baseline 的 call sites 清单（不只是看 def 本身）。本次的修法是 §5.5 「静默合丢」检测脚本 `scripts/audit_merge_loss.sh` 应当扩展支持的场景。

3. **AGENTS.md prompt-caching 铁律**优先级**高于** §3.6 任何"取本地"判断。如果一个本地特性违反 AGENTS.md 红线，无论历史多久、多 load-bearing，都应该取上游/删掉，而不是保留。
